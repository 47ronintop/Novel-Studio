# PRODUCT PRD — Novel Studio
Version: 1.0 | Status: Draft for Review | Phase: 1 Product Design

## 1. 文档目的

本文定义 Novel Studio v1 的产品目标、用户对象、核心场景、功能范围、非目标、验收标准与阶段风险。本文不定义最终技术架构、数据 Schema、UI 组件细节或任务拆分；这些内容分别在 Phase 2-6 的对应文档中完成。

本文受 `PROJECT_CONSTITUTION.md` 约束，优先引用：P1、P2、P3、P4、P5、P6、P7、P8、P9、P10。

## 2. 产品定位

Novel Studio 是面向长篇小说创作者的 project-based AI 创作 IDE。它不是聊天窗口，不是一键生成小说工具，也不是绑定单一模型供应商的内容生成器。它的核心价值是把小说创作中的资料、世界观、人物、情节、大纲、章节、版本历史、上下文策略、Prompt、Agent 和工作流组织成一个可长期演进、可迁移、可审计的本地项目。

v1 聚焦桌面端长篇小说创作，优先解决个人作者在长项目中最痛的四件事：

- 创作材料分散，长期项目难维护
- AI 输出缺乏上下文控制，容易遗忘设定或破坏风格
- Prompt、Agent 和模型配置不可控、不可版本化
- 章节内容与 AI 修改缺乏可靠历史、回退和审计能力

## 3. 目标用户

### 3.1 核心用户

职业或半职业长篇小说作者。此类用户已有持续创作习惯，关心设定一致性、章节质量、长期项目维护、版本管理和可控的 AI 辅助。

### 3.2 次级用户

网文作者、剧本创作者、游戏剧情策划、漫画脚本作者。v1 不为这些场景定制专属工作流，但产品概念和数据组织不得阻碍未来扩展。

### 3.3 非目标用户

- 只想输入一句话生成完整小说的轻量用户
- 只需要普通 ChatGPT 对话的用户
- 团队实时协作写作用户
- 需要移动端或 SaaS 托管优先体验的用户

## 4. 产品原则

- 用户拥有最终创作权：AI 只提供建议、草稿、检查和改写，不能替代用户决策（P1）。
- 项目优先：所有功能围绕项目资产展开，而非围绕聊天会话展开（P2）。
- 可编辑与可回滚：Prompt、Agent、Workflow、模型配置和上下文策略都必须可编辑，关键资产须保留历史（P3）。
- 模型无关：产品功能不依赖特定模型供应商，用户可以切换模型渠道（P4）。
- 本地优先：v1 默认以本地项目文件夹为真实数据源（P5、P7）。
- 质量优先：不以快速生成内容牺牲一致性、版本历史、可审计性和架构边界（P10）。

## 5. v1 产品目标

### 5.1 必须达成

- 用户可以创建、打开、保存一个本地 Novel Studio 项目。
- 用户可以维护人物、世界观、大纲、时间线、章节、长期记忆等核心创作资产。
- 用户可以在章节编辑器中写作，并获得自动保存、崩溃恢复、版本历史和回退能力。
- 用户可以配置多个模型供应商，并为不同工作流选择模型。
- 用户可以编辑 Prompt 模板、Agent 配置和 Workflow 定义，并保留关键资产版本历史。
- 用户可以触发 AI 辅助流程，如续写、改写、审稿、一致性检查、风格检查、伏笔/钩子建议。
- 用户可以看到 AI 调用使用了哪些上下文来源、消耗多少 token、产生什么结构化结果。

### 5.2 应该达成

- 用户可以通过 Command Palette 快速执行常用命令。
- 用户可以在 Dock Layout / Split View 中同时查看章节、人物、世界观、大纲和 AI 结果。
- 用户可以回放或检查一次 AI 上下文组装过程，理解为什么模型给出当前输出。
- 用户可以导入/导出项目文件夹，并用 Git 管理项目。

### 5.3 暂不追求

- 实时多人协作。
- 官方云端托管或 SaaS 同步。
- 移动端 App。
- 内置社交发布平台。
- 插件市场、Prompt 市场、Agent 市场的完整商业化能力。

## 6. 核心用户旅程

### 6.1 创建项目

用户创建一个新项目，填写项目名称、作品类型、语言、目标字数、题材标签、默认模型配置和本地保存位置。系统生成标准项目文件结构，并初始化基础 Prompt、Agent、Workflow 模板。

成功标准：用户可以在不理解内部文件结构的情况下完成项目创建，同时项目文件夹仍保持人类可读和可迁移。

### 6.2 建立故事资产

用户逐步录入人物、世界观、地点、势力、关键事件、时间线和大纲。AI 可以辅助整理、扩写和检查冲突，但用户必须确认后才写入正式资产。

成功标准：用户能清楚区分“草稿建议”和“已确认设定”，避免 AI 未经确认污染项目事实。

### 6.3 写作章节

用户在章节编辑器中写作。系统自动保存当前编辑状态，并定期或按事件生成历史快照。用户可以查看版本差异、回退到旧版本，也可以让 AI 基于当前章节和上下文预算进行续写、改写或审稿。

成功标准：用户不会因为崩溃或误操作丢失重要文本；AI 辅助不破坏作者的最终控制权。

### 6.4 运行 AI 工作流

用户选择工作流，例如“审稿当前章节”“检查人物一致性”“生成下一章规划”。Workflow Engine 组织 Agent 顺序，Context Engine 组装上下文，LLM Adapter 调用模型，最终以结构化结果返回。

成功标准：用户可以看到工作流的输入、使用的上下文摘要、模型配置、token 成本和输出结果；Agent 间正式交接必须是结构化 JSON（P9）。

### 6.5 调整创作系统

用户编辑 Prompt、Agent 配置、Workflow 或模型参数。系统校验配置，保留版本历史，并允许回滚。

成功标准：高级用户可以把 Novel Studio 调成适合自己作品风格的创作系统，而不需要修改代码。

## 7. 功能范围

### 7.1 Project Workspace

- 新建、打开、关闭项目。
- 显示项目元信息、当前字数、章节数量、最近活动。
- 管理项目级设置，不保存明文 API Key。
- 对项目文件结构进行基础健康检查。

### 7.2 Writing Workspace

- Markdown 章节编辑器。
- Frontmatter 元信息：章节编号、标题、状态、视角人物、时间点、地点、标签。
- 自动保存和崩溃恢复。
- 章节版本历史、差异查看和回退。
- 支持章节状态：草稿、修订中、待审、完成、归档。

### 7.3 Story Knowledge Base

- 人物档案：姓名、别名、关系、目标、冲突、弧光、出场章节。
- 世界观设定：地点、势力、规则、历史、术语。
- 大纲：卷、章、事件、目标、冲突、转折。
- 时间线：事件顺序、角色参与、因果关系。
- 记忆：长期摘要、风格记忆、人工确认的创作事实。

### 7.4 AI Assistance

- 续写：基于当前章节、目标和上下文预算生成候选文本。
- 改写：按风格、节奏、视角、信息密度进行局部修改建议。
- 审稿：指出结构、动机、节奏、设定、语言问题。
- 一致性检查：检查人物、世界观、时间线、伏笔冲突。
- 钩子建议：为章节结尾或下一章提出可选悬念。
- 摘要生成：为章节、人物、事件生成可编辑摘要。

所有 AI 输出默认进入“建议态”，用户确认后才写入正式项目资产。

### 7.5 Prompt / Agent / Workflow Studio

- Prompt 模板可查看、编辑、校验、版本化和回滚。
- Agent 配置可查看、编辑、校验、版本化和回滚。
- Workflow 可查看、编辑、启用/禁用。
- 系统内置默认模板，但不得在代码中硬编码不可编辑 Prompt。

### 7.6 Model Configuration

- 支持多个模型配置档案。
- 每个配置包含 Base URL、API Key 引用、Model Name、Temperature、Max Tokens、Top P、Timeout、Penalty 预留字段。
- UI 不展示明文密钥，日志不输出明文密钥。
- 模型调用成本和 token 用量可展示。

### 7.7 Command and Navigation

- Command Palette 支持打开文件、运行工作流、切换面板、搜索资产。
- 多 Tab 支持章节、人物、世界观、大纲、设置和工作流结果。
- Split View 支持写作时并排查看设定或 AI 建议。

### 7.8 Import / Export

- v1 至少支持项目文件夹级迁移。
- 支持从 Markdown 章节导入。
- 支持导出章节 Markdown 和项目资产 JSON。

## 8. 非功能需求

### 8.1 性能

- 打开普通项目时，核心 UI 应快速可用，索引重建不得阻塞基础阅读和编辑。
- 长篇项目应支持至少 100 万字文本规模的项目组织。
- AI 上下文组装必须受预算控制，禁止无差别塞入全文（第8节）。

### 8.2 可靠性

- 正文编辑必须自动保存。
- 异常退出后必须提供恢复入口。
- `history/` 与 `memories/` 不得被清缓存误删。
- 项目文件写入应避免半写入状态，具体事务策略在 Phase 2/3 定义。

### 8.3 安全与隐私

- 默认不采集遥测。
- 用户内容仅在用户主动触发 AI 调用时发送到用户配置的模型端点。
- API Key 本地加密存储，禁止明文写入 `project.json` 或 `settings.json`。

### 8.4 可维护性

- 产品层不要求一次实现所有模型和插件能力，但必须从需求上保留 Adapter 和 Plugin 边界。
- 任何可编辑创作资产都需要明确所有权、版本策略和回滚策略。

## 9. 信息架构

Novel Studio v1 的主界面由五类产品区域组成：

- Project Navigator：项目、章节、人物、世界观、大纲、时间线、记忆、Prompt、Agent、Workflow。
- Editor Area：章节编辑、多 Tab、Split View。
- Inspector Panel：当前章节元数据、上下文来源、版本历史、AI 结果详情。
- AI Workflow Panel：运行工作流、查看进度、查看结构化结果。
- Command Palette：跨区域快速操作入口。

## 10. 目录结构要求

v1 项目应遵循宪法第4节目录结构。PRD 层面的补充要求：

- `chapters/` 存放用户正文，是核心真实数据。
- `history/` 存放不可再生的版本和恢复数据。
- `memories/` 存放人工或 AI 辅助确认后的长期记忆，不视为纯缓存。
- `cache/` 只存放可重建索引和检索缓存。
- `prompts/`、`agents/`、`workflow/` 都属于用户可编辑资产。

## 11. 高层数据流

### 11.1 写作流

用户编辑章节 → Editor 产生编辑状态 → Application 请求保存 → Repository 写入章节草稿和恢复信息 → 达到快照条件后写入 `history/` → UI 更新保存状态。

### 11.2 AI 辅助流

用户选择工作流 → Workflow 读取任务定义 → Context Engine 按预算选择章节、人物、世界观、记忆和检索结果 → LLM Adapter 调用用户配置模型 → Agent 返回结构化 JSON → 用户查看建议 → 用户确认后写入项目资产。

### 11.3 配置流

用户编辑 Prompt/Agent/Workflow/Model 配置 → Schema 校验 → 生成版本记录 → Repository 写入对应目录 → UI 提供回滚入口。

## 12. 高层模块关系

Phase 1 仅定义产品模块，不定义最终代码模块：

- Workspace：管理项目级导航、窗口和面板。
- Editor：提供章节写作、保存状态、版本入口。
- Story Assets：管理人物、世界观、大纲、时间线和记忆。
- AI Workflows：承载用户可触发的 AI 辅助能力。
- Configuration Studio：管理模型、Prompt、Agent 和 Workflow。
- History and Recovery：贯穿正文、Prompt、Agent 和 Workflow 的版本/恢复能力。
- Import Export：负责项目迁移和外部文档交换。

## 13. 设计原因

采用 project-based IDE 而不是 chat-based 产品，是因为长篇小说创作的核心问题不是一次性生成文本，而是长期维护创作资产、上下文、风格、版本和一致性。把项目文件夹作为真实数据源，能同时满足本地优先、可迁移、Git 可管理和用户拥有数据的要求（P5、P7）。

把 Prompt、Agent、Workflow 纳入产品功能，而不是作为开发者隐藏配置，是为了落实“一切皆可编辑”（P3）。把 AI 输出默认放在建议态，是为了落实用户最终决策权（P1）。

## 14. 方案优缺点

### 优点

- 长期项目可维护性强，适合长篇创作。
- 用户不被锁定在特定模型或供应商。
- Prompt、Agent、Workflow 可演进，适合高级作者形成个人创作系统。
- 本地项目文件夹便于迁移、备份和 Git 管理。

### 缺点

- 初始产品复杂度高于普通聊天应用。
- 对 Schema、版本历史、恢复机制和 UX 信息架构要求高。
- v1 需要克制范围，避免同时做完整插件市场、多端同步和多人协作。
- 用户学习成本高，需要通过默认模板和命令入口降低门槛。

## 15. 未来扩展方案

- 剧本、漫画脚本、游戏剧情：通过项目类型、资产类型和 Workflow 模板扩展，不改变核心项目模型。
- 云同步：作为可选能力叠加在本地项目之上，不改变本地真实数据源原则。
- 插件生态：先定义 manifest、权限、能力声明和 Workflow 接入点，再逐步开放市场。
- 多人协作：未来通过 RFC 评估 CRDT、锁定编辑或 Git-based 协作模型。
- 发布平台：通过插件接入，不进入核心 v1。

## 16. 风险分析

| 风险 | 影响 | 缓解方案 |
|---|---|---|
| v1 范围过大 | 延迟核心产品验证 | 将能力分为必须、应该、暂不追求，插件市场和云同步不进入 v1 |
| AI 输出污染正式设定 | 降低项目可信度 | AI 输出默认建议态，用户确认后才能写入正式资产 |
| 上下文组装不可解释 | 用户无法信任 AI 结果 | 每次 AI 调用记录上下文来源、预算、模型配置和结构化输出 |
| 多模型适配复杂 | 拖慢主流程 | v1 产品要求架构兼容多模型，具体 Provider 落地顺序由 Phase 2/ROADMAP 决定 |
| 本地数据丢失 | 破坏核心信任 | 自动保存、崩溃恢复、版本历史作为 v1 必须能力 |
| UI 信息密度过高 | 新用户难以上手 | 默认布局聚焦章节写作，复杂面板可折叠，通过 Command Palette 快速访问 |

## 17. 验收标准

Phase 1 的产品设计完成标准：

- 明确目标用户、核心场景、v1 功能范围和非目标。
- 明确 AI 辅助能力必须保持用户确认机制。
- 明确本地项目、版本历史、自动保存、崩溃恢复为产品硬需求。
- 明确 Prompt、Agent、Workflow、模型配置为用户可编辑资产。
- 明确后续 Phase 需要继续细化架构、数据、UI、Prompt、Context、Workflow、Plugin、LLM Adapter、安全和测试。
