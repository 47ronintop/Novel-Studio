# M29 功能完成度盘点与命令面板执行闭环设计

## 背景

M28 已经把部分高可见但未完成的控件改成明确禁用，降低了“点了没反应”的问题。但用户继续试用安装版后发现，软件仍有不少功能看起来存在，实际完成度不一致。此时继续做 M27 欢迎引导会过早包装半成品，因此 M29 改为先盘点真实功能完成度，并补齐一个高频入口的真实执行闭环。

## 用户目标

用户需要一个能逐步变成正式软件的桌面写作工具，而不是只靠欢迎页解释尚未完成的功能。进入 M29 后，用户应能看到项目明确承认哪些功能可用、哪些待补，并且至少有一个原本半成品入口变成真正可点可用。

## 范围

M29 包含两部分：

1. 功能完成度审计：新增产品化文档，列出当前可见入口的状态、M29 处置和后续建议。
2. 命令面板执行闭环：把命令面板从只读列表升级为可点击执行安全命令，并同步桌面 shell state。

M29 不实现多 tab、底部面板真实内容、插件管理 UI、时间线可视化、欢迎向导或示例项目。M27 暂缓，等核心功能更稳定后再回补。

## 架构

`CommandPalette` 保持 UI 组件职责，只负责渲染命令按钮并通过 `onCommandExecute` 把命令 ID 抛给宿主。Renderer `App` 负责调用 `api.commands.execute`，并在成功后更新 `shellState`。Application 层已有 `executeCommand` safe-command 边界，M29 不新增文件系统能力，也不扩大命令权限。

数据流：

1. 用户点击标题栏命令面板按钮或快捷键打开命令面板。
2. `CommandPalette` 展示 safe commands。
3. 用户点击命令按钮。
4. Renderer 调用 `NovelStudioApi.commands.execute(command.id)`。
5. 成功时更新 `DesktopShellState` 并关闭命令面板。
6. 失败时保持当前 UI，不显示明文异常或敏感信息。

## 测试策略

- UI 测试：静态渲染确认命令项是按钮；交互测试确认点击按钮会回传命令 ID。
- Renderer 测试：新增命令执行桥或 App handler 测试，确认 `commands.execute` 成功后同步 shell state。
- Application 既有测试继续保证未知命令被拒绝。
- 全量验证沿用项目门禁。

## 风险与取舍

命令面板仍不做搜索过滤、键盘上下选择、命令分组和错误通知。这些属于后续增强，不影响 M29 的最小执行闭环。M29 不再推进欢迎页，避免把有限时间用在解释未完成能力上。
